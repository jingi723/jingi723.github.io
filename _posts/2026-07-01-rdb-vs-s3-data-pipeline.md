## 발단, 문제 인식

처음 데이터 파이프라인을 만들 때는 수집한 데이터를 전부 RDB에 저장했다.
뉴스·가격·전처리 결과를 모두 테이블로 만들었다.

처음에는 이 방식이 가장 단순해 보였다. SQL로 바로 확인할 수 있고, 테이블만 보면
데이터가 어떻게 쌓였는지도 어느 정도 파악할 수 있었기 때문이다.

그런데 멘토님께서 이런 피드백을 주셨다.

> 학습용 마트처럼 컨슈머가 직접 소비하는 최종 데이터는 RDB에 두고,
> 그 이전 단계의 원천 데이터나 중간 산출물은 S3에 저장하는 게 좋을 것 같다.
{: .prompt-info }

처음에는 이 기준이 명확하게 와닿지 않았다. 어차피 데이터인데 RDB에 넣든 S3에 넣든
큰 차이가 있을까 싶었다.

하지만 프로젝트를 다시 생각해보니, 우리는 단순히 몇 개의 데이터를 저장하는 게 아니라
뉴스·가격 데이터·피처 생성 결과를 계속 쌓고 다시 처리해야 한다. 게다가 **전 종목
커버리지**와 **준실시간성**까지 고려해야 한다. 그러면 저장소를 고르는 기준도 단순히
"어디가 편한가"가 아니라, **이 데이터가 어떤 역할을 하는가**가 되어야 한다는 생각이 들었다.

## 배경 — 저장소를 가르는 축

피드백의 의미를 풀려면 저장소를 가르는 축 두 개와 업계 용어 하나만 잡으면 됐다.

**OLTP와 OLAP.** OLTP는 지금 발생하는 거래·요청을 한 건 단위로 정확히 처리하는 성격이고,
OLAP은 쌓인 데이터를 대량으로 스캔·집계하는 성격이다. 대표 저장소도 각각 행 지향 RDB와
열 지향 분석 DB로 갈린다. 우리 파이프라인이라면 기간별 가격 흐름을 비교하거나 팩터를
계산하려고 전 종목을 훑는 작업은 OLAP에 가깝고, 특정 종목의 설명 결과를 한 건 내려주는
건 OLTP에 가깝다. 중요한 건 **뉴스 원문·원천 가격·정제 결과·분석 산출물이 모두 성격이
다르다**는 점이다. 전부 같은 방식으로 저장하면 처음엔 편하지만 데이터가 많아질수록
역할이 섞인다.

**ETL과 ELT.** 원천을 가공해서 적재하면(ETL) 저장소는 깔끔하지만 원천이 남지 않아 로직이
바뀌면 처음부터 다시 만들어야 한다. 원천을 그대로 먼저 적재하면(ELT) 재처리 여지가 생긴다.
S3에 원천을 남겨두고 나중에 다시 처리하겠다는 발상은 사실상 ELT다.

**Lake와 Warehouse.** 형식을 가리지 않고 원천·중간을 쌓아두고 읽을 때 스키마를 입히는
(schema-on-read) 쪽이 **Data Lake**이고, 적재 시점에 스키마가 정해지고(schema-on-write)
정형 조회에 최적화된 쪽이 **Data Warehouse**·마트다.

세 개를 겹쳐 보면 그림이 나온다. 원천과 중간 산출물은 schema-on-read로 S3에 남겨 재처리에
열어두고, 최종 마트는 정형 저장소에 모아 컨슈머가 바로 읽게 한다. 이게 멘토님 피드백의
실체였다.

## 그래서, 나누는 기준

개념을 정리하고 나니 멘토님 피드백의 핵심은 "RDB를 쓰지 말자"가 아니라 **RDB와 S3의
역할을 나누자**는 것이었다. 내가 이해한 기준은 이렇다.

> 다시 처리할 수 있어야 하는 데이터는 **S3**에 둔다.
> 누군가 바로 소비해야 하는 최종 데이터는 **RDB**에 둔다.
{: .prompt-tip }

![재처리 가능성으로 S3와 RDB를 나누는 결정 기준](/assets/img/posts/rdb-vs-s3-data-pipeline/storage-decision.png)
_처음 그렸던 결정 트리. 질문("다시 처리해야 하는가")은 지금도 유효하지만, **안에 든 예시는 틀렸다.** 팩터와 피처를 S3에 넣어놨는데 실제로는 RDB다. 왜 틀렸는지는 바로 아래에 적었다. 그림은 다시 그릴 예정이다._

**S3에는 원천 데이터와 정제 결과를 둔다.** 뉴스·가격·재무제표·공시 원본과, 그걸 표준
스키마로 정규화한 결과다. 이 데이터들은 로직이 바뀌면 다시 처리될 가능성이 높다. 정규화
규칙이나 품질 기준이 바뀌면 과거 구간을 다시 만들어야 하는데, 원본이 남아 있으면 외부
API를 다시 부르지 않고 정제만 다시 돌리면 된다.

**RDB에는 소비되는 데이터를 둔다.** 팩터, 분석 산출물, 설명 결과처럼 분석 엔진과 서빙이
바로 읽는 데이터다. 여기서 하나 짚어둘 게 있다. 처음에 나는 "소비자"를 API 서버로만
좁게 읽었다. 그런데 **분석 엔진도 소비자다.** 팩터와 피처는 중간에 있는 것처럼 보이지만
분석 엔진이 곧바로 읽는 입력이라, 재처리 대상이 아니라 소비 대상이다. 이걸 중간 산출물로
분류해 S3에 두면 읽는 쪽이 매번 파일을 긁어야 한다.

기준이 바뀐 게 아니라 기준을 읽는 눈이 바뀐 셈이다. "바로 소비하는 데이터는 RDB"는
그대로고, 소비자가 누구인지를 좁게 본 게 문제였다.

## 우리 프로젝트에 적용하면

우리 파이프라인은 수집 → 정제 → 분석 → 서빙으로 흐른다. 위 기준으로 나누면 경계가
정제와 분석 사이에 생긴다.

![데이터 파이프라인 아키텍처 — 수집·정제·분석·서빙 단계와 저장 계층](/assets/img/posts/rdb-vs-s3-data-pipeline/pipeline-architecture.png)
_단계 흐름(수집 → 가공 → 분석 → 서빙)은 맞지만 **저장 계층이 틀렸다.** 저장소를 넷으로 그려놨는데 실제로는 S3 둘(raw·canonical)과 RDB 하나다. 분석 결과를 S3에 그려둔 것도 틀렸다. 아래 텍스트가 맞는 구조다. 그림은 다시 그릴 예정이다._

```
S3 레이크
├─ raw/         원본 그대로. run_id 별 append, 중복 제거 없음
│               뉴스·가격·재무제표·공시
└─ canonical/   정규화 + 품질 게이트 통과분. 정체성 키로 멱등 병합, run_id 없음

RDB
                팩터, 분석 산출물, 설명 실행·결과 — 분석 엔진과 서빙이 읽는 데이터
```

`raw`는 받은 그대로 남긴다. 재무나 ETF 스냅샷은 매일 다시 받으면 같은 내용이 날마다
쌓이지만, 중복 제거를 여기서 하지 않는다. 정체성 판정을 원본까지 끌어올리면 원본이
원본이 아니게 되기 때문이다. 중복 제거와 정정은 `canonical`이 맡는다.

`canonical`에는 run_id가 없다. 같은 원본을 몇 번 정제해도 결과가 같아야 해서, 정체성
키로 멱등하게 병합한다. 뉴스라면 원문 URL 해시가 그 키다. 소스가 FMP든 BigKinds든
같은 기사면 같은 키로 합쳐진다.

이렇게 나누면 정제 로직에 문제가 생겨도 외부 API를 다시 부를 필요가 없다. `raw`를
기준으로 정제만 다시 돌리면 된다. 반대로 분석 엔진은 중간 과정을 몰라도 RDB만 읽으면
된다.

즉, **S3는 파이프라인을 다시 실행할 수 있게 해주는 저장소**이고, **RDB는 결과를
안정적으로 소비하게 해주는 저장소**다.

경계가 말로만 있는 건 아니다. 수집·정제를 담당하는 배치 워커에는 **DB 드라이버가 아예
없다.** 의존성이 S3 클라이언트와 parquet 라이브러리뿐이라, 워커가 RDB에 직접 쓰고 싶어도
쓸 수가 없다. 역할 분리를 문서가 아니라 의존성이 강제하는 셈이다.

## 이 기준으로 안 갈리는 것들

기준이 깔끔해 보이지만, 계층을 채우면서 오히려 애매해진 데이터가 있다.

**팩터**가 그렇다. 계산 로직이 바뀌면 과거 구간을 다시 만들어야 하니 재처리 대상이고,
동시에 분석 엔진이 바로 읽는 입력이니 소비 대상이다. 기준 두 개가 한 데이터에 다
걸린다. 지금은 RDB에 두고 다시 계산해서 덮는 쪽으로 잡았다. 재처리에 필요한 건 팩터
자체가 아니라 그 입력인 원본·정제 데이터인데 그건 이미 S3에 있으니, 팩터를 파일로 또
남길 이유가 없다고 봤다. 그렇다고 이게 기준에서 저절로 도출된 건 아니다.

그리고 팩터를 RDB에 두는 건 앞의 용어와 어긋난다. 전 종목을 훑어 팩터를 계산하는 건
OLAP에 가깝다고 해놓고, 정작 결과는 행 지향 RDB에 넣고 있다. 용어대로면 컬럼형이 맞는
자리다. 그럼에도 RDB를 택한 건 현재 규모에서 충분하기 때문이고, 전 종목 스캔이 실제로
느려지는 시점에 옮기는 게 맞다. 이건 기준에서 나온 결정이 아니라 **규모를 이유로 미뤄둔
결정**이라는 걸 적어둔다.

## 결론

처음에는 데이터를 RDB에 잘 저장하는 것이 파이프라인을 잘 만드는 것이라고 생각했다.
하지만 저장소를 고르는 기준은 조회가 편한지가 아니라, 그 데이터가 다시 처리될 데이터인지
바로 소비될 데이터인지였다.

> 다시 처리해야 할 가능성이 있는 데이터는 S3에 남기고,
> 바로 소비해야 하는 최종 데이터는 RDB에 저장한다.
{: .prompt-tip }

전 종목 커버리지와 준실시간성을 생각하면, 처음부터 저장소의 역할을 나눠두는 편이 나중에
파이프라인을 확장할 때 훨씬 덜 복잡할 것 같다. 다만 위에서 적었듯 이 기준이 모든 데이터를
갈라주지는 않는다. 애매한 건 애매한 채로 두고, 실제로 걸리는 시점에 다시 정하려 한다.
