## 발단, 문제 인식

처음 데이터 파이프라인을 만들 때는 수집한 데이터를 전부 RDB에 저장했다.
뉴스 데이터도 테이블로 만들고, 가격 데이터도 테이블로 만들고, 전처리 결과도 다시
테이블로 저장하는 식이었다.

처음에는 이 방식이 가장 단순해 보였다. SQL로 바로 확인할 수 있고, 테이블만 보면
데이터가 어떻게 쌓였는지도 어느 정도 파악할 수 있었기 때문이다.

그런데 멘토님께서 이런 피드백을 주셨다.

> 학습용 마트처럼 컨슈머가 직접 소비하는 최종 데이터는 RDB에 두고,
> 그 이전 단계의 원천 데이터나 중간 산출물은 S3에 저장하는 게 좋을 것 같다.
{: .prompt-info }

처음에는 이 기준이 명확하게 와닿지 않았다. 어차피 데이터인데 RDB에 넣든 S3에 넣든
큰 차이가 있을까 싶었다.

하지만 프로젝트를 다시 생각해보니, 우리는 단순히 몇 개의 데이터를 저장하는 게 아니라
뉴스·가격 데이터·피처 생성 결과를 계속 쌓고 다시 처리해야 한다. 게다가 **전 종목
커버리지**와 **준실시간성**까지 고려해야 한다. 그러면 저장소를 고르는 기준도 단순히
"어디가 편한가"가 아니라, **이 데이터가 어떤 역할을 하는가**가 되어야 한다는 생각이 들었다.

그래서 기준을 제대로 잡기 전에, 관련된 개념부터 정리해봤다.

## 배경 개념 1 — OLAP과 OLTP

먼저 저장소의 성격을 가르는 가장 큰 축, OLTP와 OLAP을 정리했다.

| 구분 | OLTP | OLAP |
|------|------|------|
| 목적 | 현재 발생하는 거래·요청 처리 | 쌓인 데이터의 조회·분석 |
| 예시 | 주문, 결제, 회원 정보 수정 | 기간별 가격 흐름, 감성 점수·주가 상관 분석 |
| 데이터 단위 | 작은 단위가 자주 생성·변경 | 대량 데이터를 한 번에 스캔·집계 |
| 핵심 가치 | 정확한 한 건을 안정적으로 처리 | 많은 데이터를 빠르게 분석 |
| 대표 저장소 | RDB (행 지향) | 분석 DB·데이터 웨어하우스 (열 지향) |

우리 파이프라인에 대입하면, 기간별 가격 흐름을 비교하거나 모델 학습을 위해 전 종목
피처를 조회하는 작업은 OLAP에 가깝다. 반면 API 서버가 특정 종목의 최종 결과 한 건을
빠르게 내려주는 건 OLTP에 가깝다.

여기서 중요한 건, **뉴스 원문·원천 가격·정제 결과·감성 분석 결과·피처 생성 결과가
모두 성격이 다르다**는 점이다. 어떤 데이터는 나중에 다시 처리해야 하고, 어떤 데이터는
최종 컨슈머가 바로 읽어야 한다. 이 둘을 같은 방식으로 저장하면 처음엔 편하지만
데이터가 많아질수록 역할이 섞인다.

## 배경 개념 2 — ETL과 ELT (가공을 언제 하나)

두 번째로 걸린 건 "그럼 가공은 언제 하지?"였다.

- **ETL** (Extract → Transform → Load): 추출한 데이터를 **가공한 뒤** 저장소에 적재.
  적재되는 건 이미 정제된 데이터라 저장소가 깔끔하지만, 원천이 남지 않으면 로직이
  바뀌었을 때 다시 만들기 어렵다.
- **ELT** (Extract → Load → Transform): 원천을 **먼저 그대로 적재**하고, 필요할 때
  저장소 안에서 가공. 원천이 남아 있어 재처리에 유리하다.

S3에 원천을 남겨두고 나중에 로직이 바뀌면 다시 처리하겠다는 접근은 사실상 ELT에
가깝다. "원천은 일단 S3에 그대로 → 가공은 뒤에서 여러 번"이라는 그림이 자연스럽게 나온다.

## 배경 개념 3 — Data Lake / Warehouse / Lakehouse

세 번째로, S3와 RDB를 나누는 게 사실 업계에서 부르는 이름이 있다는 걸 알게 됐다.

- **Data Lake**: 형식을 가리지 않고 원천·중간 데이터를 그대로 쌓는 저장소.
  읽을 때 스키마를 입히고(schema-on-read), 대용량을 싸게 보관. → **S3**가 여기에 해당.
- **Data Warehouse**: 분석·조회에 최적화된 정형 저장소. 적재 시 스키마가 정해지고
  (schema-on-write) 빠른 조회에 강함. → 최종 **마트/서빙 DB**가 여기에 가까움.
- **Lakehouse**: 레이크 위에 웨어하우스의 테이블·품질 관리를 얹어 둘을 합치려는 방향.

지금 우리가 하려는 "원천·중간은 S3, 최종 마트는 RDB"는 결국 **레이크(S3) + 서빙
마트(RDB)** 로 역할을 나누는 것이고, 이게 멘토님 피드백의 실체였다.

## 그래서, 나누는 기준

개념을 정리하고 나니 멘토님 피드백의 핵심은 "RDB를 쓰지 말자"가 아니라 **RDB와 S3의
역할을 나누자**는 것이었다. 내가 이해한 기준은 이렇다.

> 다시 처리할 수 있어야 하는 데이터는 **S3**에 둔다.
> 누군가 바로 소비해야 하는 최종 데이터는 **RDB**에 둔다.
{: .prompt-tip }

![재처리 가능성으로 S3와 RDB를 나누는 결정 기준](/assets/img/posts/rdb-vs-s3-data-pipeline/storage-decision.png)
_저장소는 "조회가 편한가"가 아니라 "이 데이터가 어떤 역할을 하는가"로 정한다._

**S3에는 원천 데이터와 중간 산출물을 둔다.** 뉴스 원문, 크롤링 결과, 원천 가격 데이터,
정제된 뉴스, 감성 분석 결과, 보정된 가격 데이터 등이다. 이 데이터들은 로직이 바뀌면
다시 처리될 가능성이 높다. 감성 분석 모델이 바뀌면 과거 뉴스 전체를 다시 돌려야 할 수
있고, 가격 보정 방식이 바뀌면 특정 기간을 다시 만들어야 할 수 있다. 그래서 파일 형태로
남겨두는 편이 유연하다.

**RDB에는 최종 컨슈머가 바로 읽는 데이터를 둔다.** 학습용 피처 마트, 날짜별 종목 피처,
모델 입력 데이터, 예측 결과처럼 구조가 명확하고 조회 대상이 분명한 데이터다. 학습
파이프라인이나 API 서버는 중간 과정이 어떻게 구성되는지 몰라도 된다. 최종적으로 정리된
마트 테이블만 안정적으로 읽을 수 있으면 된다.

## 우리 프로젝트에 적용하면

우리 파이프라인은 수집 → 가공 → 분석 → 서빙으로 흐르고, 각 단계가 저장 계층에 결과를
쓴다. 이 저장 계층을 위 기준으로 나눠보면 이렇게 정리된다.

![데이터 파이프라인 아키텍처 — 수집·가공·분석·서빙 단계와 저장 계층](/assets/img/posts/rdb-vs-s3-data-pipeline/pipeline-architecture.png)
_수집·가공·분석 결과는 S3 계층(Raw/가공/분석 결과)에, 컨슈머가 읽는 최종 데이터는 서비스 조회 DB(RDB)에._

- **Raw 저장소 (S3)** — 원천 뉴스, 원천 가격 데이터
- **가공 데이터 저장소 (S3)** — 정제 뉴스, 보정 가격, 종목 매핑, 감성 분석 결과
- **분석 결과 저장소 (S3)** — 팩터, 피처 후보 등 파생 결과
- **서비스 조회 DB (RDB)** — 학습용 피처 마트, 예측 결과 등 최종 소비 데이터

이렇게 나누면 각 단계의 책임이 명확해진다. 감성 분석 로직에 문제가 생겨도 원천 뉴스부터
다시 수집할 필요 없이 S3에 저장된 뉴스를 기준으로 다시 처리하면 된다. 피처 생성 로직이
바뀌어도 가공 데이터를 기준으로 마트만 다시 만들 수 있다. 반대로 학습 컨슈머는 복잡한
중간 과정을 신경 쓰지 않고 RDB의 최종 마트만 읽으면 된다.

즉, **S3는 파이프라인을 다시 실행할 수 있게 해주는 저장소**이고, **RDB는 최종 데이터를
안정적으로 소비하게 해주는 저장소**다.

## 결론

처음에는 데이터를 RDB에 잘 저장하는 것이 파이프라인을 잘 만드는 것이라고 생각했다.
하지만 저장소의 역할을 나눠보면서, 중요한 기준은 따로 있다는 걸 알게 됐다.

데이터를 어디에 저장할지는 조회가 편한지로 결정할 문제가 아니었다. 그 데이터가 원천인지,
중간 산출물인지, 최종 컨슈머가 직접 읽는 데이터인지에 따라 저장 위치가 달라져야 했다.
이번 프로젝트에서는 다음 기준으로 가져가려고 한다.

> 다시 처리해야 할 가능성이 있는 데이터는 S3에 남기고,
> 바로 소비해야 하는 최종 데이터는 RDB에 저장한다.
{: .prompt-tip }

전 종목 커버리지와 준실시간성을 생각하면, 처음부터 저장소의 역할을 나눠두는 편이 나중에
파이프라인을 확장할 때 훨씬 덜 복잡한 방향일 것 같다.
