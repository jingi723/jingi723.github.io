## 이 글의 범위

edge는 외부에서 뉴스·가격·재무제표·공시를 수집해 분석이 쓸 수 있는 형태로 정규화해야
한다. 첫 목표는 실시간 수집이 아니었다. **하루에 한 번 실행되고, 정해진 데이터를 처리한
뒤 종료되는 배치**를 먼저 완성하는 것이었다.

이 글에서는 그 배치의 처리 언어로 Python을, 실행 모델로 ECS Task와 Step Functions를
고른 과정을 정리한다. 데이터를 어디에 저장할지는 [이전 글](/posts/rdb-vs-s3-data-pipeline/)에
적었으니 다시 설명하지 않는다. 요약하면 원본과 정제 결과는 S3 레이크에 남고, 이 파이프라인은
거기까지만 한다.

## 요구사항

추상적으로 "확장성"이라고 적으면 나중에 아무 결정도 못 한다. 그래서 구체적으로 적었다.

| 요구사항 | 왜 |
| --- | --- |
| 하루 한 번 실행 | 데이터가 실시간으로 들어오는 구조가 아니다. 미국 장 마감 후 그날치를 받는다 |
| 실행 후 종료 | 항상 떠 있을 이유가 없다 |
| 소스별 처리 | 뉴스·가격·재무·공시가 벤더마다 API도 스키마도 다르다 |
| 실패 지점 확인 | 어느 소스에서 깨졌는지 알아야 한다 |
| 기간 재실행 | 장애나 누락 시 과거 날짜를 다시 처리해야 한다 |
| 파싱 생산성 | HTML·JSON·XML을 계속 다뤄야 한다 |

## 워커를 Python으로 둔 이유

워커가 하는 일은 외부 API 호출, HTML·JSON·XML 파싱, 표 추출, 정규화, parquet 적재다.
웹 요청을 받아 오래 떠 있는 서버가 아니라 **입력 범위를 다 처리하면 끝나는 프로세스**다.

DART 공시가 대표적이다. euc-kr로 인코딩된 HTML 테이블에서 공급계약 본문을 파싱하고,
사업보고서의 사업부문 표를 뽑아내야 한다. 여기에 붙은 의존성이 이렇다.

```
beautifulsoup4   공시 본문 HTML 테이블 파싱
pandas + lxml    사업부문 표 추출 (read_html)
html5lib         lxml이 XML-ish 공시표를 못 읽을 때 폴백
pyarrow          canonical parquet 적재
boto3            레이크 S3 백엔드
```

이걸 Java로 다시 조립할 이유를 찾지 못했다.

솔직히 고민은 있었다. 백엔드 개발자인데 워커를 Python으로 쓰면 Java와 Spring 경험을
어디서 보여주나 싶었다. 처음엔 같은 워커를 Spring으로 다시 구현할까도 생각했다.

그런데 동작하는 코드를 기술 스택을 보여주려고 다시 쓰는 건 프로젝트의 문제를 푸는 일이
아니다. 그건 이력서의 문제지 파이프라인의 문제가 아니었다. 워커는 Python으로 두기로 했다.

## ECS Service가 아니라 ECS Task

이 작업은 하루에 한 번 시작해서 정해진 데이터를 처리하고 끝난다. 메시지를 기다리는
Consumer도 아니고 요청을 받는 API 서버도 아니다. 항상 떠 있는 ECS Service보다, 필요할 때
생기고 끝나면 사라지는 **standalone Task**가 작업 모델과 맞았다.

다만 이건 모든 데이터 처리를 Task로 하겠다는 뜻은 아니다. 나중에 이벤트를 계속 소비해야
한다면 그 Consumer는 Service로 두는 게 자연스럽다. 지금 Task가 맡는 건 일 배치와, 앞으로
생길 백필·재처리 경로다.

## Scheduler가 Task를 직접 부르면 안 되었나

EventBridge Scheduler는 ECS Task를 직접 실행할 수 있다. Task가 하나뿐이면 Step Functions는
군더더기다. 그래서 처음엔 그렇게 갈 생각이었다.

그런데 수집 대상을 세어보니 8개였다.

```
FMP        뉴스 · 가격 · 재무 · ETF
BigKinds   뉴스
KIS        가격
DART       재무 · 공시
```

이걸 순차로 돌리면 느린 벤더 하나가 전체를 잡아먹는다. 그리고 하나가 실패했을 때 나머지를
살릴지 말지를 어딘가는 판단해야 한다. Scheduler에는 그런 걸 넣을 자리가 없다. 분기도,
상태도, 조건도 없다.

그래서 실행 흐름을 Step Functions에 뒀다. **워커는 데이터를 처리하고, Step Functions는
무엇을 어떤 순서로 실행할지 관리한다.**

Standard와 Express 중에서는 Standard다. Express는 실행 시간이 5분으로 제한되고 `.sync`
통합을 지원하지 않는다. ECS Task가 끝날 때까지 기다려야 하는 구조라 Express로는 아예
안 된다.

## 만들면서 알게 된 것

### 실패가 두 종류다

Task가 실패하는 방식은 하나가 아니었다. 첫 실행들이 줄줄이 깨지면서 알았다.

처음 만난 건 이거였다.

```
ResourceInitializationError: unable to pull secrets or registry auth:
  failed to fetch secret .../fmp/api-key-... from secrets manager:
  ResourceNotFoundException: can't find the specified secret value
  for staging label: AWSCURRENT

exitCode = None   lastStatus = STOPPED
```

**컨테이너가 아예 안 떴다.** 시크릿을 못 읽어서 초기화 단계에서 죽었다. 그래서
`exitCode`가 `None`이다 — 종료 코드가 있으려면 프로세스가 시작은 했어야 한다. 이건
Task 레벨 실패라 `.sync`가 알아서 잡아준다. `Catch`로 떨어진다.

문제는 다른 종류다. **Task는 정상적으로 떴고 컨테이너가 non-zero로 죽은 경우**, `.sync`는
이걸 실패로 안 볼 수 있다. Task를 띄우고 STOPPED까지 가는 게 이 통합의 책임이지, 그 안에서
프로세스가 뭘 하고 죽었는지는 다른 층의 이야기라서다.

AWS 문서도 여기까지는 말해주지 않는다. `runTask`가 HTTP 200을 주면서 `Failures` 필드가
비어 있지 않을 때 어떻게 되는지는 명시돼 있는데(그건 `.sync`가 `AmazonECS.Unknown`으로
실패시킨다), **컨테이너 종료 코드에 대해서는 문서가 침묵한다.** 그래서 확인할 방법이
없었다.

확실한 건 하나다. `.sync`의 출력 JSON에 `Containers[0].ExitCode`가 그대로 들어온다.
문서가 안 알려주면 직접 물어보면 된다. 그래서 브랜치마다 Choice를 하나씩 붙여 종료 코드를
깐다. 12개 전부에. 상태 개수가 늘어나는 건 감수했다. 실패를 못 잡는 것보다는 낫다.

![브랜치 하나가 실패로 끝나는 두 가지 길](/assets/img/posts/initial-batch-pipeline/branch-failure-paths.png)
_Task가 못 뜨면 `Catch`가, 컨테이너가 죽으면 Choice가 받는다._

### 재시도를 어느 층에 둘 것인가

처음엔 Step Functions의 `Retry`를 쓸 생각이었다. 그런데 워커의 HTTP 클라이언트에 이미
재시도가 있었다.

```python
RETRY_BACKOFF_SEC = [1, 2, 4]

for backoff in [0, *RETRY_BACKOFF_SEC]:
    ...
```

둘 다 켜면 3번 × 3번 = 최대 9번 호출된다. 외부 API 입장에서는 그냥 공격이다.

그래서 층마다 책임을 갈랐다.

```
워커 HTTP        5xx · 네트워크 · 타임아웃 → 3회 재시도 (1s, 2s, 4s)
                 4xx · 429 → 재시도 없이 즉시 중단
Step Functions   Retry 없음. 브랜치 실패 판정과 흐름만
Scheduler        SFN 실행 자체를 못 띄웠을 때만
```

429에 재시도를 안 하는 건 직관과 반대다. 하지만 429는 "너무 많이 불렀다"는 뜻이라
다시 부르면 더 맞을 뿐이다. 4xx도 마찬가지로 재시도가 고쳐줄 수 있는 실패가 아니다.
재시도가 의미 있는 건 5xx와 네트워크 오류뿐이었다.

### raw가 다 성공해야 정제로 넘어간다 — 임시로

지금은 수집 8개가 전부 성공했을 때만 정제로 넘어간다. 그런데 이건 옳아서가 아니라
**아직 만드는 중이라 그렇다.** 그리고 이 게이트는 틀렸다.

소스들은 서로 독립이다. BigKinds 뉴스가 실패한 것과 KIS 가격 정제를 돌릴 수 있는지는
아무 상관이 없다. 그런데 지금 구조는 둘의 실패를 묶어놨다. 뉴스 하나 때문에 멀쩡히 받아온
가격이 canonical로 못 올라간다. 나중에 가격 변동을 이벤트로 발행하게 되면 더 나빠진다 —
뉴스가 실패했다고 가격 이벤트를 안 보내는 건 말이 안 된다.

배치 간격이 짧아지면 더 안 맞는다. 하루에 한 번이라 다음 실행까지 24시간을 기다려도
버틸 만한 거지, 간격이 줄면 전량 성공을 기다리는 게이트는 그냥 병목이다.

그럼에도 지금 이대로 둔 건 손해가 작기 때문이다. 정제는 특정 run이 아니라 레이크 전체를
훑는다. 그래서 어제 절반만 받다 죽은 raw도 오늘 성공한 실행이 같이 승격시킨다. 게이트는
**실행 안에서의 순서 제어**지 데이터 격리가 아니라서, 데이터가 사라지지는 않고 늦어질
뿐이다.

즉 지금 구조는 게이트의 조악함을 전체 스캔으로 메우고 있다. 둘은 세트다.

### 그런데 전체 스캔도 오래 못 간다

정제는 매번 `raw/` 전체를 나열하고 전부 다시 읽는다. raw는 run_id별로 계속 append되고
절대 줄지 않으니, 어제 것도 지난달 것도 매일 다시 읽는다는 뜻이다. 지금은 raw가 작아서
안 보이지만 시간이 갈수록 선형으로 느려진다.

멱등성을 전체 스캔으로 사고 있는 셈이다. 어느 run에서 돌든 결과가 같은 건 매번 전부 다시
보기 때문이다. 정체성 키로 병합하니 같은 걸 몇 번 봐도 결과는 하나로 수렴한다. 맞는
동작이지만 비싸다.

그래서 다음에 손볼 곳이 두 군데이고, 둘은 붙어 있다.

- **게이트를 소스별로 끊는다.** 뉴스 실패가 가격을 막지 않게.
- **정제를 이번 run의 raw만 보게 한다.** 전체 스캔을 걷어내고 증분으로.

문제는 이 둘을 같이 하면 안전망이 사라진다는 것이다. 지금은 실패한 런의 raw를 다음 성공
실행이 주워가는데, run 스코프로 좁히면 아무도 안 줍는다. 실패한 수집의 raw는 고아가 된다.

여기는 자동화하지 않기로 했다. 실패는 이미 SNS로 알람이 오고 로그에 남으니, 고아가 생기면
사람이 보고 해당 run을 수동으로 다시 돌린다. 커서를 남기거나 주기적으로 전체를 스윕하는
방법도 있지만, 하루 한 번 도는 배치에 그만한 기계를 붙일 이유를 아직 못 찾았다. 재실행
손잡이(`--input-run-id`, `--from`/`--to`)는 이미 워커에 있다.

다만 `--input-run-id`는 지금 **재검증만 하고 canonical을 쓰지 않는다.** 전체 런만
authoritative라고 정해뒀기 때문이다. 수동 재실행으로 고아를 승격시키려면 이 의미부터
바꿔야 한다.

## 그래서 초기 파이프라인은 이렇게 되었다

![초기 배치 파이프라인 State Machine](/assets/img/posts/initial-batch-pipeline/pipeline-statemachine.png)
_수집 8개가 동시에 돌고, 전부 성공해야 정제 4개로 넘어간다._

수집과 정제의 개수가 안 맞는 게 보일 것이다. 재무는 정제 스텝이 아직 없어서 raw로만
남긴다. ETF 구성종목은 다르다 — **정제 스텝은 이미 있는데 State Machine에 안 걸려 있다.**
FMP·KRX 두 벤더의 이형 스키마를 공통 구성종목 fact로 정규화하는 코드가 워커에 들어 있고
CLI로는 돌아가는데, 스케줄에는 없다. KRX 수집 브랜치도 마찬가지다. 그래서 지금 ETF는
raw까지만 쌓이고 canonical로 승격되지 않는다.

워커에 스텝을 추가하는 것과 그게 실제로 돌게 하는 건 별개였다. 실행 흐름을 Step Functions에
둔 대가이기도 하다 — 스텝은 파이썬으로 늘리는데 배선은 테라폼에서 해야 하니, 한쪽만 하고
넘어가기 쉽다. 다음에 붙일 예정이다.

스케줄이 뉴욕 시간인 건 지금이 PoC라서다. 초기에는 FMP에서 미국 데이터를 받는 게 가장
쉬워서 거기로 먼저 뚫었고, 스케줄도 미국 장 마감에 맞췄다. MVP의 대상은 국내 ETF이고,
그때는 cron과 타임존만 바꾸면 된다. 실행 시점을 Scheduler에 따로 떼어둔 덕에 State
Machine도 워커도 손댈 게 없다.

각 브랜치는 ECS Fargate Task 하나를 띄우고, Task는 Python 워커를 CLI로 실행한다.

```
data-pipeline ingest-raw --source bigkinds --run-id 20260715T...
data-pipeline normalize-news --run-id 20260715T...
```

역할은 이렇게 나뉜다.

```
Scheduler         언제 실행할지
Step Functions    무엇을 어떤 순서로, 실패를 어떻게 판정할지
ECS Task          워커의 일회성 실행 환경
Python 워커        수집 · 파싱 · 정규화
S3 레이크          raw와 canonical
```

Step Functions에 데이터를 넘기지는 않는다. `run_id`만 넘기고 워커가 레이크에서 직접
읽는다. 뉴스 본문을 상태 사이로 나르면 payload 제한에 걸리고, 무엇보다 나를 이유가 없다.

정제 Task는 시크릿이 없는 task definition을 빌려 쓴다. 정제는 벤더 API를 안 부르고 레이크만
읽고 쓰기 때문에 API 키가 필요 없고, 모든 task definition이 같은 role을 공유한다. 새로
만들 이유가 없어서 안 만들었다.

| | **합계** | **5,907** | **5,885** |

숫자보다 재밌는 건 안 맞는 칸들이다.

**FMP 뉴스가 166건 받아 144건 저장했다.** 22건이 준 건 아니고, 같은 기사가 여러 종목
질의에 걸린 것이다. 기사를 버리지 않고 "이 기사는 이 종목으로도 걸렸다"는 mention만
기존 레코드에 합친다. 어느 종목으로 걸렸는지는 질의한 쪽만 아는 정보라 raw에서 보존해야
한다.

**DART 공시가 0건인데 성공이다.** 그날 공시가 없었다. 0건과 실패는 다르다 — 이걸 구분
못 하면 조용한 실패를 성공으로 읽게 된다.

**BigKinds가 1,295건 전부 저장했는데 그 안에 절단이 둘 있다.** 삼성전자와 SK하이닉스가
페이지 상한(5)에 걸려 창이 잘렸다. 로그에는 `kind: truncation`으로 남았지만 브랜치는
성공이다. 자주 나는 절단으로 파이프라인 전체를 세우지 않기로 한 결과다. 대신 잘렸다는
사실은 로그에 남아서, 구간을 좁혀 다시 돌리면 채울 수 있다.

**ETF 구성종목을 2,629건 받았다.** 이 실행의 이름이 `verify-etf`인 이유다. 그런데
앞에서 적었듯 ETF는 정제 브랜치가 State Machine에 없다. **저 2,629건은 지금 raw에만
있고 canonical에 없다.** 수집이 되는 걸 확인했으니 다음은 배선이다.

## 아직 해결하지 않은 것

**처리 단위가 run 전체다.** 워커는 `run(storage, run_id)` 형태로 한 번에 한 실행분을
처리한다. 실행 방식과 처리 로직이 분리돼 있긴 하다 — CLI는 인자만 파싱하고, 스텝은
storage를 주입받아서 ECS도 Step Functions도 모른다. 백엔드가 `local`이면 로컬 디렉터리에
그대로 돌아간다.

하지만 문서 하나를 처리하는 함수는 없다. 나중에 이벤트를 메시지 단위로 소비해야 한다면
지금 코드로는 부를 게 없다. "향후 재사용 가능하게 분리했다"고 말하기엔 절반만 맞다.

**같은 날짜를 두 번 돌리면 어떻게 되는가.** raw는 run_id별로 계속 쌓이도록 일부러 그렇게
만들었다. 중복 제거는 정제가 맡는다. 그런데 정제가 정말 멱등한지, 두 번 돌린 결과가 한 번
돌린 것과 같은지는 아직 제대로 확인하지 않았다.

**게이트를 소스별로 끊고, 정제를 증분으로 바꿔야 한다.** 위에 적은 대로 둘은 세트다.
전체 스캔을 걷어내는 순간 실패한 런의 raw를 주울 사람이 없어지니, 알람과 수동 재실행이
실제로 굴러가는지부터 확인해야 한다. `--input-run-id`가 canonical을 쓰도록 바꾸는 것도
여기 딸린 일이다.

**ETF 구성종목 정제와 KRX 수집을 State Machine에 붙여야 한다.** 코드는 이미 있다.

다음 글에서는 이걸 다룬다. 같은 배치를 반복 실행해도 결과가 흔들리지 않는지, 어디서
정체성을 판정해야 하는지.
